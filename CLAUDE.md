# InsightConnect Plugin Development

This repository contains Rapid7 InsightConnect plugins. Each plugin lives in its own subdirectory.

## Quick Reference

| Task | Command |
|------|---------|
| Scaffold a new plugin | `insight-plugin create` (after writing `plugin.spec.yaml`) |
| Regenerate schemas after spec change | `insight-plugin refresh` |
| Lint and validate plugin | `insight-plugin validate` |
| Run unit tests | `python -m pytest unit_test/ --cov=icon_<name> --cov-report=term-missing` |

## Plugin Naming Convention

- Directory name: `my_plugin` (snake_case)
- Python package name: `icon_my_plugin` (always prefixed with `icon_`)
- Entrypoint binary: `bin/icon_my_plugin`

## Python Version

All plugins target **Python 3.11**.

## Dependency Rules

All entries in `requirements.txt` must be **version-pinned** (e.g., `requests==2.31.0`). No `>=` or bare package names.

## Auto-Generated Files — Never Edit

These files are regenerated by `insight-plugin refresh` and any manual changes will be overwritten:
- `icon_*/actions/*/schema.py`
- `icon_*/triggers/*/schema.py`
- `icon_*/tasks/*/schema.py`
- `icon_*/connection/schema.py`
- `setup.py`
- `bin/icon_*`

## Testing Requirements

- **Minimum 80% code coverage** on all new and modified code
- Unit tests live in `unit_test/`
- JSON fixture files for mock responses live in `unit_test/responses/`
- Use `parameterized` for data-driven tests
- Mock `requests.Session.request` — not the Connection class itself
- Test all HTTP error codes the API can return, not just the happy path

## Versioning

Follow semver strictly. Every version bump requires a `CHANGELOG.md` entry.

| Change | Bump |
|--------|------|
| Remove/rename input or output field, change field type, add required input | **Major** |
| New action/trigger/task, add optional field | **Minor** |
| Bug fix, SDK update, dependency update | **Patch** |

## Specialist Agent

For hands-on plugin work, use the `@insightconnect` agent. It has full knowledge of:
- Plugin directory structure and file roles
- `plugin.spec.yaml` schema and all field types
- Connection, Action, Trigger, and Task implementation patterns
- `PluginException` and `ConnectionTestException` usage
- Unit test patterns with `MockResponse`, fixture files, and `parameterized`
- The `insight-plugin` CLI workflow
- Versioning rules and CHANGELOG format

Invoke it with: `@insightconnect <your request>`
