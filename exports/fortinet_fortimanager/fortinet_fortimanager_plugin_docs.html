Fortinet FortiManager Plugin

Rapid7 InsightConnect · v1.0.0 · Supported: FortiManager 7.0.x, 7.2.x, 7.4.x

Description

Fortinet FortiManager is a centralized network security management platform that provides single-pane-of-glass administration for multiple FortiGate devices. This plugin enables InsightConnect SOAR workflows to manage address objects, address groups, and firewall policies through FortiManager's JSON-RPC API.

Key Features

  • Create, delete, and list address objects
  • Add and remove address objects from address groups
  • Check if an address exists in an address group
  • Retrieve firewall policies from policy packages
  • Install policy packages to managed FortiGate devices
  • Supports API Token and session-based authentication
  • Per-action ADOM override for multi-tenant environments
  • Built-in RFC 1918 and whitelist filtering for address creation

Requirements

  • Fortinet FortiManager 7.0 or later
  • An API token (FortiManager 7.2.2+ recommended) or admin username and password
  • Network connectivity from the InsightConnect orchestrator to FortiManager on HTTPS (port 443)
  • JSON-RPC API access enabled for the admin account

Setup & Configuration

FortiManager Side — API Token Authentication (Recommended)

API token authentication is available in FortiManager 7.2.2 and later. This is the recommended approach as it avoids session management overhead and is more suitable for automation.

  1. Navigate to System Settings > Admin > Administrators
  2. Click Create New > REST API Admin
  3. Configure the admin:
    • Username: Choose a descriptive name (e.g., insightconnect-api)
    • Admin Profile: Assign the least-privilege profile you created (see Permissions section below)
    • JSON API Access: Set to Read-Write
    • Trusted Hosts: Restrict to the InsightConnect orchestrator IP(s) for security
    • ADOM: Assign to the specific ADOM(s) this integration will manage
  4. Save — FortiManager will generate an API token. Copy and store it securely; it cannot be retrieved later.
Tip: If you are running FortiManager 7.0.x or 7.2.0–7.2.1, API token auth is not available. Use Session-Based authentication instead.

FortiManager Side — Session-Based Authentication (Legacy)

For FortiManager versions below 7.2.2, use a standard admin account with JSON-RPC access:

  1. Navigate to System Settings > Admin > Administrators
  2. Create or edit an admin user (e.g., insightconnect-session)
  3. Set the Admin Profile to the least-privilege profile (see Permissions below)
  4. Under the admin's settings, ensure JSON API Access is set to Read-Write
  5. Set Trusted Hosts to the InsightConnect orchestrator IP(s)
  6. Save
Security Note: Session-based auth transmits credentials with each login request. Always use HTTPS and restrict trusted hosts to minimize exposure.

Network Requirements

  • InsightConnect orchestrator must be able to reach FortiManager on TCP/443 (HTTPS)
  • If using a non-standard port (e.g., 8443), append it to the hostname: fortimanager.example.com:8443
  • If FortiManager uses a self-signed certificate, set SSL Verify to false (not recommended for production)
  • Ensure any intermediate firewalls allow the orchestrator's IP to reach FortiManager's management interface

Permissions & Least Privilege

FortiManager uses Admin Profiles to control access. Create a dedicated profile for the InsightConnect integration at System Settings > Admin > Admin Profiles > Create New. Below are recommended least-privilege configurations for different use cases.

Profile: Read-Only (Audit / Visibility)

Use this profile when workflows only need to query objects and policies — no modifications.

Permission CategoryAccess LevelRationale
Policy & ObjectsRead-OnlyRequired to list address objects, groups, and policies
Administrative DomainRead-OnlyRequired to access ADOM-scoped objects
System SettingsNoneNot needed for object queries
Device ManagerNoneNot needed for object queries
JSON API AccessRead-WriteMust be Read-Write for any API access (FortiManager requirement)

Supported actions: Get Address Objects, Check if Address in Group, Get Policies

Profile: Object Management (Block/Unblock Workflows)

Use this profile for threat response workflows that create/delete address objects and manage group membership — the most common SOAR use case.

Permission CategoryAccess LevelRationale
Policy & ObjectsRead-WriteRequired to create, delete, and modify address objects and groups
Administrative DomainRead-OnlyRequired to access ADOM-scoped objects
System SettingsNoneNot needed for object management
Device ManagerNoneNot needed unless installing policy packages
JSON API AccessRead-WriteRequired for all API operations

Supported actions: All object actions (Get, Create, Delete, Add to Group, Remove from Group, Check if in Group, Get Policies)

Profile: Full Automation (Object Management + Policy Install)

Use this profile when workflows need to push changes to FortiGate devices after modifying objects. This is the most privileged level required by the plugin.

Permission CategoryAccess LevelRationale
Policy & ObjectsRead-WriteRequired for all object and policy operations
Administrative DomainRead-OnlyRequired to access ADOM-scoped objects
System SettingsNoneNot needed by the plugin
Device ManagerRead-WriteRequired for Install Policy Package action
JSON API AccessRead-WriteRequired for all API operations

Supported actions: All actions including Install Policy Package

Important: The Install Policy Package action pushes configuration to production FortiGate devices. Only grant Device Manager Read-Write if workflows actively deploy policy changes. Consider using a separate connection/profile for install operations to reduce blast radius.

ADOM Scoping

For multi-tenant environments, assign the API admin to specific ADOMs rather than granting global access:

  • When creating the admin, select only the ADOM(s) the integration needs to manage
  • An ADOM-scoped admin cannot see or modify objects in other ADOMs
  • Use the per-action adom parameter to target specific ADOMs at runtime

Get Address Objects

Retrieve address objects from a FortiManager ADOM with optional filtering by name, subnet, or FQDN.

Input

NameTypeRequiredDescription
Name FilterstringOptionalFilter address objects by name (case-insensitive exact match)
Subnet FilterstringOptionalFilter by subnet value (e.g., 198.51.100.100/32)
FQDN FilterstringOptionalFilter by FQDN value (e.g., example.com)
ADOMstringOptionalOverride connection ADOM

Output

Returns a list of address objects matching the specified filters.

[
  {
    "name": "MaliciousHost",
    "type": "ipmask",
    "subnet": "198.51.100.100/32"
  }
]

Create Address Object

Create a new address object in a FortiManager ADOM. Automatically detects address type (IP, CIDR, FQDN) and applies whitelist/RFC 1918 filtering.

Input

NameTypeRequiredDescription
AddressstringRequiredIP address, CIDR notation, or FQDN
Address Object NamestringOptionalCustom name. Defaults to the address value if not provided
Whitelist[]stringOptionalAddresses that should not be blocked (IPs, CIDRs, domains)
Skip RFC 1918booleanRequiredSkip creation for private IP addresses. Default: true
ADOMstringOptionalOverride connection ADOM

Output

Returns a boolean indicating success and the created object details.

{
  "success": true,
  "address_object": {
    "name": "MaliciousHost",
    "type": "ipmask",
    "subnet": "198.51.100.100/32"
  }
}

Delete Address Object

Delete an address object from a FortiManager ADOM by name.

Input

NameTypeRequiredDescription
Address ObjectstringRequiredName of the address object to delete
ADOMstringOptionalOverride connection ADOM

Output

{
  "success": true
}

Add Address Object to Group

Add an address object to an address group. The object must already exist in the ADOM. Addition is idempotent — adding an object that already exists in the group succeeds without error.

Input

NameTypeRequiredDescription
Address ObjectstringRequiredName of the address object to add
GroupstringRequiredName of the target address group
ADOMstringOptionalOverride connection ADOM

Output

{
  "success": true,
  "address_objects": ["MaliciousHost", "AnotherHost"]
}

Remove Address Object from Group

Remove an address object from an address group. The group must have at least one remaining member after removal.

Input

NameTypeRequiredDescription
Address ObjectstringRequiredName of the address object to remove
GroupstringRequiredName of the address group
ADOMstringOptionalOverride connection ADOM

Output

{
  "success": true,
  "address_objects": ["AnotherHost"]
}

Check if Address in Group

Check if an address exists in an address group. Supports matching by object name or by searching the stored value (subnet/FQDN) of each member.

Input

NameTypeRequiredDescription
AddressstringRequiredAddress to check (matched by name or value depending on Enable Search)
GroupstringRequiredName of the address group to check
Enable SearchbooleanRequiredWhen true, searches member stored values instead of names. Default: false
ADOMstringOptionalOverride connection ADOM

Output

{
  "found": true,
  "address_objects": ["198.51.100.100/32"]
}

Install Policy Package

Install a policy package to managed FortiGate devices or device groups. This pushes the current ADOM configuration (including any address object changes) to the specified targets.

Caution: This action deploys configuration changes to production devices. Ensure changes are validated before triggering this in automated workflows.

Input

NameTypeRequiredDescription
Policy PackagestringRequiredName of the policy package to install
Target Devices[]stringOptionalList of target device names (at least one device or group required)
Target Device Groups[]stringOptionalList of target device group names
ADOMstringOptionalOverride connection ADOM

Output

Returns the task ID for the asynchronous install operation. Use FortiManager's task monitor to track progress.

{
  "task_id": 512
}
