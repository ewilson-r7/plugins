Description

Kiro is an AI-powered development environment that assists with building, testing, reviewing, and releasing InsightConnect plugins. The .kiro/ directory in this repository provides Kiro with the context, rules, and workflows it needs to operate effectively within the plugin development lifecycle.

This configuration transforms Kiro from a general-purpose AI into a specialized InsightConnect plugin engineer that understands the SDK, follows conventions, avoids known pitfalls, and can execute multi-step build and release workflows autonomously.

How It Works

The .kiro/ configuration operates as a three-layer system:

1. Steering (Context Rules)

Steering files provide persistent context — facts, conventions, and constraints that Kiro should always (or conditionally) keep in mind. They don't tell Kiro what to do; they tell it how to behave when doing anything. Steering files are loaded based on their inclusion mode: automatically on every interaction, when specific file patterns are open, or only when manually activated.

2. Skills (Step-by-Step Workflows)

Skills are multi-step procedures that Kiro follows when performing complex tasks. Each skill defines a sequence of phases — research, implement, validate — that Kiro executes in order. Skills reference steering files for their rules but add orchestration on top: deciding which commands to run, what order to do things, and when to stop.

3. Hooks (Event Automation)

Hooks trigger automatically in response to events — a user prompt, a completed task, or a file change. They run skills or custom logic without the user needing to ask. This creates a feedback loop: fixes update steering to prevent recurrence, and completed builds automatically regenerate documentation.

These three layers interact: steering provides the rules, skills orchestrate the work, and hooks automate the triggers. A typical flow might be: a hook detects a completed build → triggers the doc-generation skill → which follows steering rules about formatting and content.

Steering Files

Steering files inject context into Kiro's reasoning. They define what the AI should know and remember, organized by concern. Each file has an inclusion mode that determines when it loads into context.

repos.md

auto

Repository routing table mapping plugin names to their development and production locations. Includes dev workflow commands (insight-plugin CLI), the decision tree for choosing between prod and dev repos, and standard directory paths.

plugin-dev.md

auto

High-level product overview of InsightConnect and the plugin SDK. Covers the core rules every plugin must follow, the version numbering scheme (semver table for major/minor/patch), key CLI commands, and the relationship between spec files and generated code.

common-mistakes.md

auto

A catalog of 38 anti-patterns observed in past plugin development sessions. Each entry describes the mistake, why it's wrong, and the correct approach. Also includes a pre-submission update checklist to verify before marking work as complete.

structure.md

fileMatch: plugins/**

Project layout conventions, directory naming rules (snake_case, icon_ prefix), and the three standard workflows: new plugin creation, adding actions/triggers, and updating existing plugins. Loaded whenever any file inside plugins/ is open.

implementation.md

fileMatch: plugins/**/*.py

Detailed code patterns for connection classes, action/trigger implementations, API client utilities, pagination strategies, rate-limit handling, and input/output guards. Loaded when editing Python source files within plugins.

exceptions.md

fileMatch: plugins/**/*.py

PluginException preset reference (Cause, Assistance, Data fields), ConnectionTestException patterns, and rules for when to raise vs. return errors. Ensures consistent error messaging across all plugin actions.

plugin-spec.md

fileMatch: **/plugin.spec.yaml

Complete field reference for plugin.spec.yaml — all supported types, enum syntax, credential access patterns, action/trigger/task definitions, and version_history formatting. The authoritative guide for spec authoring.

testing.md

fileMatch: **/unit_test/**

Unit test conventions: mock strategies (patching requests at the session level), test file structure, fixture organization in responses/ and payloads/ directories, coverage requirements, and parameterized test patterns.

prospector.md

manual

Resolution guide for linter issues surfaced by Prospector. Covers mccabe complexity reduction, bandit security findings (B101, B105, etc.), pylint refactoring suggestions, and how to apply fixes without breaking plugin conventions.

git-workflow.md

manual

Production release branch strategy for creating upstream PRs to rapid7/insightconnect-plugins. Covers branch naming, sync procedures, PR creation, and the review process for getting plugins into production.

quick-actions-create.md

manual

Guide for creating InsightConnect analyst quick actions — lightweight, importable workflow snippets that analysts can run on-demand from the InsightConnect interface.

quick-actions-update.md

manual

Procedure for updating existing quick actions after a plugin version bump. Covers updating version references, re-testing, and resubmitting to the workflows repository.

Skills

Skills define multi-step workflows that Kiro executes in sequence. Each skill is a complete procedure — from research through validation — that produces a specific deliverable.

plugin-dev

The master router skill. Classifies user intent (build, review, release, etc.), determines the correct repository (production vs. development), and delegates to the appropriate specialized skill. This is the entry point for most plugin-related requests.

plugin-build-prep

Pre-build readiness checks. Verifies that tooling is installed and functional (insight-plugin, Python, Docker), resolves the latest SDK version from the upstream repository, confirms the correct Python version, and ensures the working directory is clean before starting a build.

create-new-plugin

Full new plugin workflow from zero to validated. Researches the target API, writes the plugin.spec.yaml, scaffolds the project with insight-plugin create, implements all actions and the connection, writes unit tests with mocked responses, and runs validation.

create-plugin-action

Adds a new action to an existing plugin. Edits the spec to add the action definition, bumps the version, runs insight-plugin refresh to generate scaffolding, implements the action logic, adds API methods if needed, writes tests, and validates.

plugin-release

Production release workflow. Syncs the local master with upstream, creates a properly-named release branch, stages changes, creates the PR against rapid7/insightconnect-plugins, and guides through the review process.

plugin-review

Read-only code review against the master upstream repository. Pulls the latest conventions and known issues, then analyzes the specified plugin for conformance, anti-patterns, missing tests, spec inconsistencies, and documentation gaps without modifying any files.

refactor-plugin-auth

Auth migration workflow for converting plugins from user-delegated authentication to app-only (client_credentials) OAuth2 flows. Updates the connection spec, rewrites the auth logic, updates token handling, and verifies the migration with connection tests.

generate-plugin-html-docs

Generates self-contained HTML documentation for a plugin. Reads the spec and implementation, then produces a single-file doc page with sidebar navigation, setup instructions, permissions tables, action I/O documentation, custom types, and version history — matching the style of this docs site.

workflow-builder

Builds importable .icon workflow files from plain-language descriptions. Translates natural language into the InsightConnect workflow JSON format with proper step wiring, variable references, and trigger configurations.

workflow-submission

Submits completed workflows to the rapid7/insightconnect-workflows repository. Validates the workflow structure, creates the PR with proper metadata, and ensures the submission meets the upstream repository's contribution requirements.

Hooks

Hooks run automatically in response to events, creating feedback loops that keep the configuration up to date and documentation fresh without manual intervention.

update-steering-on-fix

User-triggered

When activated, reviews the current session for mistakes that were caught and corrected. Extracts the lesson learned and updates the relevant steering file (typically common-mistakes.md) to prevent the same mistake from recurring in future sessions.

regenerate-docs

PostTaskExec

Triggers after a development build completes successfully. Regenerates the HTML documentation for the affected plugin and updates the docs/index.html navigation if a new plugin was added. Ensures docs always reflect the latest spec and implementation.

Settings

mcp.json

MCP (Model Context Protocol) server configuration. Currently configures a fetch server that allows Kiro to retrieve web content — used for reading API documentation, checking upstream repository state, and resolving latest SDK versions from GitHub.

permissions.yaml

Shell command allowlists that control what the agent can execute. Defines permitted commands (like insight-plugin, git, python, docker) and blocks potentially dangerous operations. Acts as a security boundary for autonomous execution.

Inclusion Modes

Each steering file declares an inclusion mode that determines when it's loaded into Kiro's context:

ModeBehaviorUse Case
auto Always loaded on every interaction Core rules that apply universally — routing, conventions, common mistakes
fileMatch Loaded when a file matching the glob pattern is open in the editor Context-specific rules — Python patterns when editing .py files, spec rules when editing YAML
manual Only loaded when explicitly activated by the user or a skill Specialized guides used infrequently — linter fixes, release procedures
Auto-mode files consume context budget on every request, so they should be kept concise and high-signal. FileMatch files provide relevant depth without the cost. Manual files are unlimited in length since they're only loaded on demand.

For Contributors

If you're new to this repository and want to understand the AI-assisted workflow:

  • Start with steering/plugin-dev.md — gives you the product context and core rules
  • Read steering/structure.md — explains the project layout and standard workflows
  • Review steering/common-mistakes.md — learn from past mistakes before making new ones
  • Check skills/plugin-dev.md — understand how the master routing skill delegates work
Steering files are living documents. If you discover a new anti-pattern or convention, add it to the relevant file. The update-steering-on-fix hook can do this automatically during development sessions.

When adding new steering files, choose the most restrictive inclusion mode that still covers your use case. Prefer fileMatch over auto when the content is only relevant to specific file types.
