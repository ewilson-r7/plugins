Active Directory LDAP

AD LDAP (Active Directory Lightweight Directory Access Protocol) is an application protocol for querying and modifying items in Active Directory. This plugin queries Microsoft's Active Directory service to programmatically manage and query an Active Directory environment.

Description

The Active Directory LDAP plugin enables InsightConnect workflows to interact with on-premises or hybrid Active Directory environments using LDAP. It supports user lifecycle management (create, enable, disable, unlock, delete), group membership changes, password resets, object modifications, and flexible LDAP queries.

The plugin supports both NTLM and Kerberos (SASL GSSAPI) authentication, with an Auto mode that attempts Kerberos first and falls back to NTLM. It also supports SSL/TLS and channel binding for secure communication.

Key Features

  • Add and remove user accounts to automate provisioning/deprovisioning of users
  • Disable and enable user accounts to contain security risks
  • Reset user passwords when a user forgets their login information
  • Modify user groups to add or remove users from custom and built-in groups
  • Run a custom LDAP query to retrieve, add, modify, or delete Active Directory objects
  • Kerberos (SASL GSSAPI) authentication support for ticket-based auth
  • Channel binding support for enhanced security
  • Bulk enable/disable operations for multiple user accounts

Requirements

  • Host name and port number (the default TCP/UDP port for LDAP is 389, and 636 for LDAP over SSL)
  • Administrative credentials (username and password)
  • To connect, you must have NTLM or Kerberos credentials
  • For NTLM, enter credentials with the DOMAIN\username format
  • For Kerberos, provide the domain name and KDC address. The plugin handles ticket acquisition automatically
  • Network connectivity from the InsightConnect orchestrator to the Active Directory domain controller
  • SSL is required for password reset operations

Setup

Option A: NTLM Authentication (On-Premises AD)

Step 1: Prepare a Service Account

  1. Open Active Directory Users and Computers on a domain controller
  2. Create a new user account for InsightConnect (e.g., svc_insightconnect)
  3. Set a strong password and check Password never expires (or manage rotation via your organization's policy)
  4. Assign the account to the appropriate groups based on the actions you need (see Permissions section below)
Security: Use a dedicated service account for InsightConnect. Do not use a personal admin account. Store credentials securely and rotate them according to your security policy.

Step 2: Verify Network Connectivity

  • Ensure the InsightConnect orchestrator can reach your domain controller on port 389 (LDAP) or 636 (LDAPS)
  • If using a firewall, allow TCP traffic on the appropriate port
  • For multi-domain environments with chase referrals enabled, ensure connectivity to all referenced domain controllers

Step 3: Configure the Connection in InsightConnect

FieldTypeRequiredDescription
HoststringRequiredServer hostname, e.g. example.com. Do not include protocol prefix — the plugin adds ldap:// or ldaps:// automatically.
PortintegerRequiredPort number. Default: 389. Use 636 for LDAPS.
Use SSLbooleanRequiredEnable SSL/TLS. Required for password reset and add user operations.
Use Channel BindingbooleanOptionalRequire a secure TLS channel before binding. Enable if your AD enforces channel binding. Default: false.
Authentication Typestring (enum)RequiredChoose NTLM, Kerberos, or Auto. Auto tries Kerberos first, then NTLM. Default: Auto.
Username and PasswordcredentialRequiredFor NTLM: use DOMAIN\username format. For Kerberos: use the account's username.
KerberosobjectOptionalKerberos configuration: kdc (IP of domain controller) and domain_name (e.g. example.com). If KDC is not set, the Host value is used.
Chase ReferralsbooleanRequiredFollow referrals to other AD servers in multi-domain environments. Default: true.
Tip: The plugin uses NTLM or Kerberos authentication depending on your selection. For NTLM, the username must include the domain prefix (e.g., MYDOMAIN\svc_insightconnect). For Kerberos, the plugin acquires a ticket automatically from the KDC using the provided credentials.

Option B: Kerberos Authentication

Step 1: Prepare a Service Account

  1. Create a service account in Active Directory (same as NTLM setup above)
  2. Ensure the account has a valid SPN (Service Principal Name) or is able to authenticate against the KDC
  3. Note the fully qualified domain name (e.g., corp.example.com)
  4. Note the IP address of the KDC (typically the domain controller, e.g., 10.0.1.11)

Step 2: Configure the Connection

  1. Set Authentication Type to Kerberos (or Auto to try Kerberos first)
  2. In the Kerberos field, provide:
    • kdc: IP address of your domain controller / KDC
    • domain_name: Your AD domain (e.g., example.com)
  3. Provide the service account credentials in the Username and Password field
How it works: When Kerberos is selected, the plugin configures /etc/krb5.conf at runtime, acquires a Kerberos ticket using kinit, and binds to LDAP using SASL GSSAPI. The ticket is managed automatically — no manual keytab or ticket management is needed.

Option C: Microsoft Entra ID (Azure AD) — Hybrid Setup

If your environment uses Microsoft Entra ID (formerly Azure AD) with on-premises AD synchronization, you can connect via the on-premises domain controller. For scenarios that require direct Entra ID Graph API access, consider the Microsoft Entra ID Admin plugin instead.

For hybrid environments using this LDAP plugin:

  1. Connect to the on-premises domain controller that syncs with Entra ID
  2. Changes made via LDAP will sync to Entra ID through Azure AD Connect
  3. Use NTLM or Kerberos authentication to the on-premises DC

Troubleshooting

  • 401 Unauthorized / Invalid Credentials — Verify username format. NTLM requires DOMAIN\username. Kerberos requires correct KDC and domain name.
  • Connection timeout — Verify network connectivity to the domain controller on the configured port. Check firewall rules.
  • SSL required error — Password reset and add user (with password) require SSL. Set Use SSL to true and port to 636.
  • Chase referrals failure — In multi-domain environments, ensure all referenced DCs are reachable from the orchestrator.
  • Channel binding errors — If your AD enforces channel binding (via LDAP Channel Binding policy), enable the Use Channel Binding option.
  • Special characters in DN — Objects containing = or * must be escaped with backslash (e.g., CN=Robert \= Bob Smith).

Permissions

Active Directory uses a delegation model for permissions. The service account used by InsightConnect needs specific permissions depending on which actions you intend to use. Below are recommended permission profiles organized by use case.

Profile 1: Read-Only (Queries Only)

Use this profile if you only need to search and query AD objects without making changes.

Permission / GroupAccess LevelActions Enabled
Domain Users (default)ReadQuery, Query Group Membership
By default, any authenticated domain user can read most AD attributes. No additional delegation is needed for read-only queries.

Profile 2: User Management

Use this profile for user lifecycle automation — enabling, disabling, unlocking accounts, and resetting passwords.

Permission / GroupAccess LevelActions Enabled
Account OperatorsCreate/Modify/Delete users and groupsAdd User, Enable User, Enable Users, Disable User, Disable Users, Unlock User, Delete, Modify Groups
Reset Password (delegated)Reset password on target OUsReset Password, Force Password Reset
Write userAccountControlModify account flagsEnable User, Disable User
Write lockoutTimeClear lockout attributeUnlock User
Important: SSL must be enabled for password reset operations. Without SSL, the Reset Password and Force Password Reset actions will fail. The Add User action will create the user but cannot set a password or enable the account without SSL.

Profile 3: Object Management

For workflows that modify AD object attributes, move objects between OUs, or manage group memberships.

Permission / GroupAccess LevelActions Enabled
Write All Properties (on target OUs)Modify any attributeModify Object
Delete Subtree (on target OUs)Delete objectsDelete
Move/Delete Child ObjectsMove objects between OUsMove Object
Write Member attribute (on groups)Modify group membershipAdd or Remove Object from Group

Profile 4: Full Automation

For comprehensive automation workflows that need all capabilities. Use the Domain Admins or Account Operators group with additional delegated permissions.

Permission / GroupAccess LevelActions Enabled
Account OperatorsFull user/group managementAll actions
Domain AdminsFull domain accessAll actions (including admin accounts)
Least Privilege: Avoid using Domain Admins unless absolutely necessary. Delegate specific permissions on target OUs to minimize blast radius. Use separate service accounts for read-only vs. write operations where possible.

Delegating Permissions on Specific OUs

To apply least-privilege access, delegate permissions on specific Organizational Units:

  1. Open Active Directory Users and Computers
  2. Right-click the target OU → Delegate Control
  3. Add the InsightConnect service account
  4. Select the specific tasks to delegate (e.g., "Reset user passwords and force password change at next logon")
  5. Complete the wizard

This limits the service account's permissions to only the OUs where automation is needed.

Query

Run an LDAP query against Active Directory. Supports standard LDAP filter syntax (RFC4515) with automatic pagination for large result sets.

Input

NameTypeRequiredDescription
search_filterstringRequiredThe filter of the search request. Must conform to LDAP filter syntax (RFC4515). Example: (sAMAccountName=joesmith)
search_basestringRequiredThe base of the search request. Example: DC=example,DC=com
attributes[]stringOptionalAttributes to search. If empty, returns all attributes.

Output

Returns an array of matching objects with their DN and attributes, plus a count of results.

{
  "results": [
    {
      "dn": "CN=Joe Smith,OU=Users,DC=example,DC=com",
      "attributes": {
        "sAMAccountName": "joesmith",
        "cn": "Joe Smith",
        "memberOf": ["CN=Domain Users,CN=Users,DC=example,DC=com"],
        "userAccountControl": 512,
        "whenCreated": "2021-01-14 17:48:26+00:00"
      }
    }
  ],
  "count": 1
}

Query Group Membership

Return users and groups belonging to a specific group. Supports nested group expansion and automatic pagination.

Input

NameTypeRequiredDescription
group_namestringRequiredName of the group to check membership. Example: Domain Users
search_basestringRequiredThe base of the search request. Example: DC=example,DC=com
include_groupsbooleanOptionalInclude groups in results.
expand_nested_groupsbooleanOptionalExpand nested groups in results.

Output

Returns matching members with full attributes and a count.

{
  "results": [
    {
      "dn": "CN=Example User,CN=Users,DC=example,DC=com",
      "attributes": {
        "cn": "Example User",
        "sAMAccountName": "Example User",
        "memberOf": ["CN=Domain Users,CN=Users,DC=example,DC=com"],
        "userAccountControl": 514
      }
    }
  ],
  "count": 1
}

Add User

Add a new user account to Active Directory. SSL is recommended — without SSL the action creates the user but cannot assign a password or enable the account.

SSL Required: Without SSL enabled, this action will create the user in a disabled state without a password. Enable SSL and use port 636 for full functionality.

Input

NameTypeRequiredDescription
domain_namestringRequiredThe domain name this user will belong to. Example: example.com
logon_namestringRequiredThe logon name for the account. Example: jdoe
first_namestringRequiredUser's first name.
last_namestringRequiredUser's last name.
user_oustringRequiredThe OU the user account will be created in. Default: Users
passwordpasswordRequiredThe account's starting password.
account_disabledbooleanRequiredSet to true to disable the account at creation. Default: true
user_principal_namestringRequiredThe user principal name. Example: user@example.com
additional_parametersobjectOptionalAdditional user parameters in JSON format. Example: {"telephoneNumber":"(617)555-1234"}

Output

Returns a boolean indicating success or failure.

{ "success": true }

Input
